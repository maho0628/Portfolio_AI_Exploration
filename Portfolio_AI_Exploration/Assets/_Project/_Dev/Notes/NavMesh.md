① NavMesh は「移動ロジック」じゃなくて「移動可能領域の定義」

NavMesh はキャラクターを動かす仕組みではなく、
「この世界のどこが歩けるか」を定義するためのものだと理解しました。

緑のエリア = 歩行可能領域

キャラの知能ではなく「地形ルール」

AI設計の土台

👉
「まず世界を定義する → その上でAIを乗せる」
という考え方に切り替わった。

② Agent Radius は“見た目”じゃなく“設計値”

Agent Radius はキャラクターの当たり判定というより、
世界との距離感をどう設計するかの値だと学びました。

半径が大きいと通路が通れなくなる

小さいと不自然に密着する

見た目とロジックは必ずしも一致しない

👉 Prototype シーンで
「通れる / 通れない」境界を確認してから本番に反映するようにした。

③ 障害物で穴が空く = 設計が正しく作用している証拠

障害物によって NavMesh に穴が空くことを確認し、
環境側の設定が正しく AI に影響していることを検証しました。

NavMeshObstacle の挙動確認

Bake結果を見て判断

「動いたからOK」ではなく
「なぜ動いたか」を確認

④ 到達不能は「バグ」じゃなく「設計イベント」

NavMesh 上で到達不能な地点が発生すること自体は、
想定すべき挙動だと考えています。

失敗時の遷移

Idle に戻す

別行動を選択する余地

👉
AIの賢さではなく、状態管理の問題として扱う方針にした。

NavMesh の設定値は、
一度決めると世界設計全体に影響するため、
必ず Prototype で検証してから本番に転記する運用にしています。
